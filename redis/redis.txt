String类型：
    value是字符串，根据字符串的格式不同，可以分三种
        string：普通字符串
        int：整数类型，可以自增自减
        float：浮点类型。可以自增自减
        不管那种数据类型，底层都是字节数组形式存储，不过编码方式吧不同，字符串类型最大空间不超过512M

    常见命令：
        SET: 添加或者修改已存在的一个String类型键值对
        GET: 根据key获取string类型的value
        MSET: 批量添加多个string类型键值对
        MGET；批量获取多个string类型的value
        INCR：让一个整型的key自增1
        INCRBY:让一个整形的key自增并指定步长 例如incrby num 2，让num自增2
        INCRBYFLOAT:让一个浮点类型的数字自增并指定步长
        SETNX：添加一个String类型的键值对，前提是这个key不存在
        SETEX：添加一个String类型的键值对，并指定有效期
    key的结构：Redis的key允许有多个单词形成层级结构，多个单词之间用':'隔开，格式如下：
        项目名：业务名：类型：id

hash类型的常见命令：
    HSET key field value 添加或者修改hash类型key的field的值
    HGET key field 获取一个hash类型的field的值 
    HMSET：批量添加多个hash类型的field值
    HMGET：批量获取多个hash类型的field值
    HGETALL:获取一个hash类型的key中所有的field跟value值的值
    HKEYS：获取一个hash类型的key中所有的field
    HVALS:获取一个hash类型的所有value值
    HINCRBY:让一个hash类型的key字段自增并指定步长
    HSETNX:添加一个hash类型的key的field字段值，前提是这个field不存在，否则不执行、

List类型的常见命令：
    LPUSH key element...：向列表左侧插入一个或者多个元素
    LPOP key：移除并返回列表左侧的第一个元素，没有返回nil
    RPUSH key element...: 向列表右侧插入一个或多个元素
    RPOP key：移除并返回列表右侧的第一个元素
    LRANGE  key star end： 返回一段角标范围内的所有元素
    BLPOP和BRPOP：与LPOP和RPOP类似，只不过在没有元素时等待指定时间，不是直接返回nil

Set类型（无序，元素不可重复，查找快，支持交集并集差集功能）常见命令：
    SADD key member...: 向set插入一个或多个元素
    SREM key member...: 移除set中的指定元素
    SCARD key：返回set中的元素个数
    SISMEMBER key member：判断一个蒜素是否在set中
    SMEMBERS：获取set中所有的元素
    SINTER key1 key2：求key1与key2的交集
    SDIFF key1 key2：求key1与key2的差集
    SUNION key1 key2：求key1与key2的并集
    
SortedSet（可排序，元素不重复，查询速度快）常用命令：
    ZADD key score member：添加一个或者多个元素到sorted set， 如果已经存在，则更新score值
    ZREM key member：删除sorted set中的一个指定元素
    ZSCORE key member：获取sorted set中的指定元素的score值
    ZRANK key member：获取sorted set中指定元素的排名
    ZCARD key：获取sorted set中的元素个数
    ZCOUNT key min max：统计score值在给定范围的所有元素个数
    ZINCRBY key increment member：让sorted set 中的指定元素自增，步长为指定的increment值
    ZRANGE key min max：按照score排序后，获取指定排名范围内的元素
    ZRANGEBYSCORE key min max：按照score排序后，后去指定score范围内的元素
    ZDIFF、ZINTER、ZUNION：求差集交集并集

RDB：Redis数据快照，将内存所有数据记录到磁盘中，Redis故障重启后，从磁盘读取快照文件，恢复数据
    两种方式：save跟bgsave
    save：快捷命令直接敲，主进程执行，阻塞所有命令
    bgsave：fork一个子进程执行命令，但是会占用cpu资源，也可以自动调用，修改
        redis.conf文件，找到配置：save 900 1（900秒钟，有一个redis指令执行，触发bgsave）
        如果bgsave执行过程中，存在数据修改，会在内存中会拷贝一份数据给主进程进行修改和读取，
        bgsave保存的数据依旧不变（copy-on-write），避免脏写

AOF：所有命令保存到aof文件中，可以称做是命令日志文件，修改redis.conf配置文件开起aof
    appendonly yes
    appendfilename "appendonly.aof"

    appendsync always 每执行一次写命令，立刻记录aof文件
    appendsync everysec 写命令执行放入aof缓冲区，每隔1秒将数据缓冲区写到aof文件，默认方案
    appendsync no 写命令执行放入aof缓冲区,由操作系统决定何时将缓冲区内容写入磁盘

                RDB                                     AOF
持久化方式     定时对整个内存进行快照           记录每一次执行的命令
数据完整性     不完整，两次备份之间会丢失        相对完整，取决于刷盘策略 
文件大小       会有压缩，文件体积小             记录命令，文件体积很大
宕机恢复速度   很快                            慢
数据恢复优先级  低，数据完整性不如aof            高，但是数据完整性更高
系统资源占用    高，大量cpu和内存消耗           低，主要是磁盘IO资源，但AOF重写会占用大量CPU和内存资源
使用场景        可以容忍数分钟的数据丢失，追求更快的启动速度  对数据安全性要求较高常见


搭建redis集群：
    1、在/tmp目录下创建7001,7002,7003文件
    2、修改/etc/redis/redis.conf配置文件，关闭aof：appendonly no
    3、拷贝redis.conf到文件目录下：
        sudo echo 7001 7002 7003 | xargs -t -n 1 cp /etc/redis/redis.conf
        或者
        sudo cp /etc/redis/redis.conf 7001
        sudo cp /etc/redis/redis.conf 7002
        sudo cp /etc/redis/redis.conf 7003
    4、更改port跟dir，port替换成7001,7002,7003, dir替换成/tmp/7001，/tmp/7002，/tmp/7003
    5、启动3个redis服务：
        redis-server /tmp/7001/redis.conf
        redis-server /tmp/7002/redis.conf
        redis-server /tmp/7003/redis.conf

        redis-cli -p 7001
        redis-cli -p 7002
        redis-cli -p 7003
        目前3个都是独立节点
        主节点设置：
            5.0以前用slaveof，5.0以后用replicaof
            临时永久两种模式
                修改配置文件永久生效：slaveof masterip masterport
                临时生效，重启失效：  slaveof masterip masterport
            注意：7001不执行该命令
            slaveof 192.168.137.128 7001
            取消节点设置：replicaof no one
            查看节点状态：info replication
            注意：如果主节点redis有密码，需要设置密码：config set masterauth xxj240803
    
主从数据同步原理：
    第一次是全量同步，执行bgsave生成RDB文件，发送RDB文件，如果中间主节点产生新数据，
        会保存在repl_baklog文件缓存中，一起发送给从节点
    数据同步原理：
        Replication Id:简称replid，是数据集的标记，id一致则说明是同一数据集，每一个master都有唯一的replid，slave则会集成master节点的replid
        offset：偏移量，随着记录在repl_baklog中的数据增多而逐渐增大。slave完成同步时也会记录当前同步的offset。
            如果slave的offset小于master的offset，说明slave数据落后于master，需要更新。
        因此，slave做数据同步需要向master声明自己的replication Id和offset，master才可以判断需要同步哪些数据

数据同步优化：
    在master配置中配置repl-diskless-sync yes启用无磁盘复制，避免全量同步时磁盘IO（注意：开启后占用网络带宽，需要在网络好的环境下使用）
    Redis单节点的内存占用不要太大，减少RDB导致过多的磁盘IO
    适当提高repl_baklog的大小，发现slave宕机尽快恢复故障，避免全量同步
    限制一个master上的slave节点数量，如果实在多slave，可以采用主-从-从链式结构，减少master压力
